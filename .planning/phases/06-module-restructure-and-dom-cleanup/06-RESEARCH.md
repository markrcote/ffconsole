## Summary

Phase 6 is a structural refactor with no new mechanical behavior. All three requirements are satisfied by changes to three files (`index.html`, `js/app.js`, `js/ui/battle.js`) plus one new file (`js/ui/battleModal.js`). There is no external dependency, no new library, and no backend change required.

The core technical challenge is the `battle.js` DOM migration: 17 `document.getElementById()` call sites spread across `renderBattle()` and its nested closures must become `container.querySelector()` calls. The `updateStaminaBars()` helper is special ‚Äî it is called from both the main event handler and the async luck-test closure, so it needs the container threaded through to both call sites. The `loadCombatHistory()` export uses two additional global ID queries (`combat-log-toggle`, `combat-log-list`) but those elements are dynamically generated inside the history container it receives ‚Äî those queries should become `historyContainer.querySelector()` rather than `container.querySelector()`.

The `charCreate.js` pattern is the exact reference implementation. `battleModal.js` follows it verbatim: create overlay div, set `className = 'modal-overlay active'`, build inner HTML with the full combat structure, append to `document.body`, call `renderBattle(modalInnerDiv, ...)`, and expose `openBattleModal()` as the sole export. Close/teardown is Phase 7's job; the skeleton can export a no-op `closeBattleModal()` stub or nothing.

**Primary recommendation:** Migrate `battle.js` first (it can be tested with the existing inline DOM before removing it), then create `battleModal.js`, then swap `index.html` and `app.js` last as one atomic change.

### Pattern 1: Container-Scoped DOM Queries

**What:** Replace `document.getElementById('foo')` with `container.querySelector('#foo')` throughout `battle.js`, where `container` is the argument already accepted by `renderBattle()`.

**When to use:** Any DOM query that targets elements owned exclusively by a single module's rendered output.

### Pattern 2: Modal Overlay Lifecycle (charCreate.js reference)

**What:** Create-on-open, destroy-on-close. The modal element does not exist in the DOM until opened; it is removed entirely on close.

**When to use:** Any panel that appears on demand and must not leak event listeners on repeated opens.

**Reference implementation from `charCreate.js`:**
```javascript
// Create
const overlay = document.createElement('div');
overlay.className = 'modal-overlay active';
overlay.innerHTML = `...`;
document.body.appendChild(overlay);

// Scope all queries to overlay
const someEl = overlay.querySelector('#some-id');

// Destroy
function cleanup() { overlay.remove(); }
```

**`battleModal.js` follows this exactly:**

### Pattern 3: D-17 Module Boundary

**What:** UI modules never import `app.js`. State flows in via `getState` function reference; mutations flow out via `callbacks` object.

**Application to this phase:** `battleModal.js` imports `renderBattle` from `./battle.js` and nothing else from the app layer. `app.js` imports `openBattleModal` from `./ui/battleModal.js` and wires the click handler.

```javascript
// app.js ‚Äî the only place that connects the button to the modal
const startBattleBtn = document.getElementById('start-battle-btn');
if (startBattleBtn) {
    startBattleBtn.addEventListener('click', () => {
        openBattleModal(
            () => ({ state, combatState, currentBook }),
            {
                onStart: startCombat,
                onRollRound: rollCombatRound,
                onFlee: endCombat,
                onEnd: endCombat,
                onStatSync: syncStateFromServer,
                onCombatEnd: () => { combatState.active = false; },
                onTestLuck: (bookNumber, luckCurrent, round, context, damageBefore) =>
                    testCombatLuck(bookNumber, luckCurrent, round, context, damageBefore),
            }
        );
    });
}
```

### Anti-Patterns to Avoid

- **Leaving historyEl as `document.getElementById`:** The `historyEl` ref at line 185 in `battle.js` must be removed ‚Äî history is not inside the modal container. `loadCombatHistory()` is called from `app.js` with its own container ref.
- **Passing `.modal-overlay` instead of `.battle-modal` as container:** `renderBattle()` queries IDs like `#combat-setup` that live inside `.battle-modal`. Passing the outer overlay means queries traverse a wider tree ‚Äî functionally fine but breaks the abstraction. Pass the inner `.modal` div.
- **Adding global IDs to modal HTML that already exist in `index.html`:** After removing `combat-section`, all IDs like `combat-setup`, `combat-active`, etc. no longer exist in `index.html`. The modal creates fresh copies ‚Äî no collision risk, but the code must never simultaneously have both in the DOM.
- **Not updating `app.js` import:** After this phase, `app.js` no longer calls `renderBattle()` directly ‚Äî it calls `openBattleModal()`. The import line must change from `import { renderBattle, loadCombatHistory }` to `import { openBattleModal } from './ui/battleModal.js'` plus keeping `loadCombatHistory` separate.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Modal overlay display | Custom CSS toggle class | Existing `.modal-overlay.active` pattern | Already in `css/style.css` lines 276‚Äì293 |
| DOM scoping helper | Custom `scopedQuery()` utility | `container.querySelector()` directly | Native DOM, no abstraction needed at this scale |
| Event cleanup on modal close | Manual `removeEventListener` lists | `overlay.remove()` (destroy-on-close pattern) | Removing the element GCs all listeners automatically |

**Key insight:** The destroy-on-close pattern (`overlay.remove()`) is specifically chosen because it eliminates listener leak tracking entirely. Every event listener attached to the overlay or its children is garbage-collected when the overlay element is removed. Phase 7 teardown requires only `overlay.remove()`.

## Common Pitfalls

### Pitfall 1: historyEl Scope Confusion

**What goes wrong:** The developer migrates all `document.getElementById()` calls to `container.querySelector()` and also moves the `#combat-history` query into the container scope. But `#combat-history` is in `index.html` (stays there per D-02), not inside the modal container.

**Why it happens:** `battle.js` currently queries `historyEl` at line 185 inside `renderBattle()` alongside all the other elements. It looks like it belongs in the same migration.

**How to avoid:** `historyEl` must be removed from `renderBattle()` entirely. History refreshes are triggered by `app.js` calling `loadCombatHistory(currentBook, historyContainer)` from outside the modal. The calls to `loadCombatHistory(currentBook, historyEl)` inside the Roll Round, Flee, and luck-test handlers (lines 421, 471, 266) must be converted to use the `onCombatEnd` callback to signal `app.js` to refresh, OR the `historyContainer` must be threaded in as an additional argument to `renderBattle()`. Recommendation: add a `historyContainer` argument to `renderBattle()` or use the existing `onCombatEnd` callback to trigger a history reload from `app.js`.

**Warning signs:** `historyEl` is null at runtime; history section stops updating after combat.

### Pitfall 2: updateStaminaBars Signature Break

**What goes wrong:** `updateStaminaBars()` is called in multiple places inside the closures of `renderBattle()`. After adding `container` as the first parameter, all call sites inside `renderBattle()` must be updated. Missing even one call site means the bars silently fail to render.

**Why it happens:** There are 4 call sites for `updateStaminaBars()` in `battle.js` (lines 244, 339, 384, 464). They are inside async closures, making them easy to overlook.

**How to avoid:** Search the file for all occurrences of `updateStaminaBars(` before and after the change to verify count matches.

**Warning signs:** Stamina bars do not update during combat; no JS errors (graceful `if (playerFill)` guards hide the failure).

### Pitfall 3: combat-log toggle queries after loadCombatHistory migration

**What goes wrong:** `loadCombatHistory()` creates elements with IDs `combat-log-toggle` and `combat-log-list`, then immediately queries them via `document.getElementById()`. After the refactor these IDs only exist inside `historyContainer`, not in the global document.

**Why it happens:** `loadCombatHistory()` is a separate export that receives its own `historyContainer` argument. Its internal toggle queries should scope to `historyContainer`, not the battle `container`.

**How to avoid:** Change lines 603‚Äì604 to `historyContainer.querySelector(...)` rather than `document.getElementById(...)`.

**Warning signs:** Battle history expand/collapse toggle stops working.

### Pitfall 4: Stale import in app.js

**What goes wrong:** After creating `battleModal.js`, `app.js` still imports `renderBattle` from `battle.js` and tries to call it directly on `.combat-section`, which no longer exists.

**Why it happens:** `app.js` lines 71‚Äì84 initialise the battle panel on the inline combat section. That section is removed in this phase.

**How to avoid:** Remove the `renderBattle` import and the `combatSection` initialisation block from `app.js`. Add `import { openBattleModal } from './ui/battleModal.js'`. Keep `loadCombatHistory` import from `battle.js` for the history section.

**Warning signs:** JS module error at load time (attempting to import from wrong path), or `combatSection` is null and the old wiring silently does nothing.

## Open Questions

1. **historyEl threading strategy**
   - What we know: `battle.js` currently queries `#combat-history` inside `renderBattle()` and passes it to `loadCombatHistory()` calls on combat end. After removing this element from the container, the module needs another way to trigger history refreshes.
   - What's unclear: Whether the planner should add `historyContainer` as an optional fourth argument to `renderBattle()`, or route refresh through `onCombatEnd` callback.

Keeps existing refresh calls in `battle.js` working with minimal change. `battleModal.js` passes `document.getElementById('combat-history')` when creating the modal. This is consistent with how `app.js` currently passes the history container.

   - What's unclear: Whether the planner prefers a no-op stub export now vs. leaving it entirely to Phase 7.
