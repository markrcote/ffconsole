- Created `js/ui/battleModal.js` following the charCreate.js create-on-open pattern

- Removed entire `<section class="mechanics-section combat-section">` block from `index.html` (enemy setup form, stamina bars, round cards, summary — all now generated by `battleModal.js`)
- Renamed "Tests" section to "Actions" (`tests-section` class -> `actions-section`, heading text updated)
- Added `#start-battle-btn` button to Actions section with `mechanic-btn mechanic-btn--primary` classes
- Updated `app.js` imports: removed `renderBattle`, added `openBattleModal` from `./ui/battleModal.js`
- Replaced inline `combatSection` initialization block with `startBattleBtn` click handler calling `openBattleModal()`
- Restored `loadCombatHistory()` call in `init()` so battle history loads on page load

## Decisions Made

- `battleModal.js` fetches `historyContainer` via `document.getElementById('combat-history')` at open time. This is intentional — the history container lives in `index.html` outside the modal, so it cannot be inside the modal's container scope. The element is stable in the DOM throughout the session.
- `closeBattleModal()` stub exported in Plan 02 so Phase 7 can `import { openBattleModal, closeBattleModal }` without changing the import statement.
- `loadCombatHistory()` call added back to `init()` in `app.js` — previously `renderBattle()` called it at init time via the historyContainer arg, but now `renderBattle()` is no longer called from `app.js` directly. The history section needs an initial load.
