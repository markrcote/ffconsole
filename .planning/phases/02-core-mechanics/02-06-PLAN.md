# Plan 6: Gap Fix — Skip localStorage When Backend Returns Empty Sessions

**Phase:** 02 — Core Mechanics
**Goal:** When the backend responds with an empty session list, skip the localStorage fallback and return null — so the app shows character creation instead of stale cached state.
**Requirements:** CHAR-01 (first-load flow)
**Depends on:** None

## Context

`load()` in `js/storage.js` only enters the backend path when `sessions.length > 0`. When the backend returns `[]` (empty array), control falls through to the localStorage fallback, which can return a stale session and show an old adventure sheet instead of the character creation modal.

The fix: if the backend responds OK, treat it as authoritative. An empty `[]` from the backend means "no sessions" — return `null` directly without consulting localStorage.

## Tasks

### Task 1: Return null when backend responds OK with empty sessions

**File:** `js/storage.js`
**Action:** Modify

In `load()`, after the `if (sessions.length > 0) { ... }` block (lines 57–75), add an explicit `return null` so an OK-but-empty backend response does not fall through to localStorage:

Current code (lines 55–79):
```javascript
        if (response.ok) {
            const sessions = await response.json();
            if (sessions.length > 0) {
                // ... build games, return data ...
            }
        }
    } catch (e) {
        console.error('Failed to load from server:', e);
    }

    // localStorage fallback
```

Change to:
```javascript
        if (response.ok) {
            const sessions = await response.json();
            if (sessions.length > 0) {
                // ... build games, return data ...
            }
            // Backend reachable but no sessions — authoritative empty state.
            // Do NOT fall through to localStorage (could be stale).
            return null;
        }
    } catch (e) {
        console.error('Failed to load from server:', e);
    }

    // localStorage fallback (only reached when backend is unreachable)
```

The `return null` goes after the closing `}` of the `if (sessions.length > 0)` block, still inside the `if (response.ok)` block.

**Verification:** `grep -n "return null" js/storage.js` — should show the new line inside the `if (response.ok)` block.

## Success Criteria

- When the backend responds `200 []`, `load()` returns `null`
- `null` from `load()` causes `init()` in `app.js` to call `showCharCreate` (existing behaviour)
- localStorage fallback is only reached when the backend request fails (network error, 5xx, etc.)
- Existing sessions still load correctly when backend returns non-empty array

## Verification Steps

1. Delete `ff.db`, restart the server, open the app — character creation modal appears immediately (no localStorage clear needed).
2. Create a character, reload — session loads correctly from backend.
3. Stop the server, reload — localStorage fallback kicks in and shows the last session (offline resilience preserved).
