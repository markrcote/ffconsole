<context>
The app is a Fighting Fantasy gamebook companion — vanilla JS frontend, FastAPI/SQLite backend, no build step, mobile-first. The user can switch devices since everything is server-backed.

Phase 1 is pure infrastructure — no user-visible changes. It unblocks all feature phases by fixing the backend schema (add mechanics_json), cleaning up the storage layer (delete legacy compat shim), setting up the config system (book-specific mechanic configs), and establishing clean JS module boundaries.

Key pitfalls to watch from research:
- Pydantic will silently strip book mechanic fields unless mechanics_json is in the schema (Pitfall 3)
- Mobile buttons need touch-action: manipulation to avoid double-tap (Pitfall 4)
- Don't let combat state leak into session state (Pitfall 1)
</context>
